{% extends "base.html" %} {# Dashboard page (Phase 3, PR #10). Server-renders the full page on first paint, then lets the user reload individual sections via HTMX. The page has two pickers (Alpine.js): * Card picker (Todas las cards + every active CreditCard row). * Period picker (current month + last 3/6/12 months + all-time). Changing either picker triggers an HTMX partial refresh for the five sections: KPI grid, categories, top merchants, monthly bars, and recurring list. The HTMX requests hit the dedicated ``/dashboard/sections/*`` endpoints (defined in :mod:`app.web.router`), which call :class:`app.services.dashboard.DashboardService` directly so the picker change is one DB roundtrip per section (not two: there is no JSON-to-DOM step). The initial paint is the same code path: the route handler calls the service once per section, then renders the same partials. The first paint is meaningful with JavaScript disabled; the Alpine + HTMX layer is a progressive enhancement. No JS chart library is loaded — every bar is a Tailwind ``
`` with ``style="width: X%"``. The bar width is computed server-side (the spec scenario "Each month renders a Tailwind bar with ``style=...``" is the contract). The test surface in :mod:`tests.test_web_phase3` asserts the ``style="width: X%"`` strings are present in the response body. #} {% block title %}{{ app_name }} — Dashboard{% endblock %} {% block content %}
